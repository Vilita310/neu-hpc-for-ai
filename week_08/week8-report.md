# Week 8 Report: Multi-GPU DeepSeekV3 MoE with NCCL

## Overview

For this assignment I implemented a multi-GPU DeepSeekV3 MoE operator in CUDA using NCCL for inter-GPU communication. The implementation uses both data parallelism (splitting input tokens across GPUs) and expert parallelism (splitting routed experts across GPUs), following the distributed MoE inference pipeline from the lecture slides.

## Implementation

### Parallelism Design

I split things across 2 GPUs like this:

- **Tokens**: each GPU gets half the input tokens (T/2 each). This is the data parallelism part.
- **Routed experts**: each GPU owns half the routed experts (NR/2 each). GPU 0 gets experts 0 through NR/2-1, GPU 1 gets NR/2 through NR-1. This is the expert parallelism part.
- **Router and shared experts**: these are just replicated on both GPUs. The router needs to see every local token to decide where it goes, and the shared experts are always active for all tokens, so replicating them is the simplest approach.

### Forward Pass

I followed the 6-phase pipeline from the lecture:

**Phase 1 — Shared experts.** Each GPU independently runs the shared experts on its local tokens and accumulates the results. No communication needed here since the shared experts are replicated.

**Phase 2 — Routing.** Each GPU runs the replicated router (linear projection → softmax → greedy top-K → normalize × scale) on its local tokens. This gives us the expert indices and weights for each token. Again no communication, everything is local.

**Phase 3 — Permutation (all-to-all dispatch).** This is where it gets interesting. Based on the routing results, each GPU needs to send tokens to the GPU that owns the assigned expert. For example, if GPU 0 has a token routed to expert 5 and GPU 1 owns experts 4-7, that token needs to go to GPU 1.

I first exchange the send counts via `ncclAllGather` so every GPU knows how much data it will receive. Then I pack each token's data along with its local expert id, routing weight, and original token index into a buffer, and use grouped `ncclSend`/`ncclRecv` to do the all-to-all exchange.

**Phase 4 — Expert computation.** After receiving tokens from the all-to-all, each GPU groups them by local expert id and batch-processes each expert's tokens through the SwiGLU MLP. This is efficient because we can batch all tokens going to the same expert into one MLP forward call.

**Phase 5 — Un-permutation (all-to-all return).** Another round of `ncclSend`/`ncclRecv` sends the expert outputs back to the GPU that originally owned each token. I pack the result vector along with the routing weight and original token index so the receiving GPU knows where to put things back.

**Phase 6 — Scaling and combination.** Each GPU now has both the shared expert output (from Phase 1) and the returned routed expert results (from Phase 5). The final output per token is:

```
output[t] = shared_expert_sum[t] + Σ_k weight_k × routed_expert_k(x[t])
```

### CUDA Kernels

I wrote a few simple CUDA kernels for the compute:

- `linear_k` — batched matrix-vector multiply for the linear projections
- `silu_mul_k` — fused SiLU activation and element-wise multiply (the SwiGLU gate)
- `softmax_k` — per-row softmax (one block per token)
- `topk_k` — greedy top-K selection with weight normalization and scaling
- `add_k` — element-wise vector addition

The MLP forward (`mlp_fwd`) chains these together: gate projection → up projection → SiLU×up → down projection, which matches the SwiGLU structure used in DeepSeek.

### Multi-GPU Setup

Each GPU runs in its own pthread. All threads share the same `ncclUniqueId` and initialize separate NCCL communicators via `ncclCommInitRank`. Each thread sets its own CUDA device, creates its own stream, and loads its partition of the model.

## What I Learned

The trickiest part was getting the all-to-all communication right. The permutation phase needs to pack tokens with enough metadata (local expert id, weight, original token index) so the receiving GPU can process them and the un-permutation phase can put results back in the right place. I spent a while debugging off-by-one issues in the buffer packing.

The 6-phase pipeline from lecture maps really cleanly to the code once you see it. Shared experts and routing are completely local (no comm), then it's all-to-all → compute → all-to-all → combine. The symmetry between permutation and un-permutation is nice.

One thing I didn't optimize is the send buffer layout — right now I'm using a fixed max-size buffer per destination which wastes some memory. In a real system you'd want variable-length buffers, but for this assignment the fixed layout keeps the NCCL send/recv simple.
